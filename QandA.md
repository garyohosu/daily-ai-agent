## Q01. Grokタスクの送信元メールアドレスは確定か？
**セクション**: 10.3, 11.2
**質問**: `From: noreply@x.ai` と記載があるが、実際に確認済みか？送信元が異なる場合、Gmailフィルタ条件全体に影響する。
**回答**: 現時点では `Grok <noreply@x.ai>` から `garyohosu@gmail.com` 宛に通知が届いた実績があるため、初期フィルタ条件は `From: noreply@x.ai` とする。ただし正式固定値とはせず、Phase 0 で再確認し、差分があれば Gmail フィルタ条件を更新する。

## Q02. GrokタスクはどのX/Grokアカウント・プランで使うか？
**セクション**: 9.1
**質問**: Grokのタスク機能はX Premium+（有料）が必要と思われるが、利用するアカウントとプランは確定しているか？また、タスクの実行時刻（06:30 JST）を指定できるUIや機能がGrok側に存在するか？
**回答**: X の課金ユーザー環境で Grok のタスク機能が利用可能であることを、実機画面で確認済み。少なくとも本プロジェクトで使用する運用アカウントではタスク機能が使える。なお、一般論としてどの課金プラン全体で使えるかまでは本仕様では保証せず、本運用アカウントで使用可能であることを要件充足条件とする。

## Q03. Grokタスクの自動実行メカニズムは何か？
**セクション**: 9.4
**質問**: 「毎朝06:30 JST実行」とあるが、Grokタスクのスケジューリングはどのように行うのか？Grok UI上のスケジュール設定か、外部スクリプトからAPIを叩くのか。
**回答**: 06:30 JST の定期実行は、Grok のタスク機能そのもので設定可能であり、実際に「毎日 6:30」の設定画面を確認済み。したがって初期版では、外部スケジューラを使わず、Grok タスクの内蔵スケジュール機能を正式採用する。

## Q04. Gmail APIの認証方式は何を使うか？
**セクション**: 11.3, 12.1
**質問**: Gmail取得にGmail APIを使う想定と思われるが、認証はOAuth2（ユーザー認証）かService Accountか？資格情報の管理・保管方法はどうするか？
**回答**: 認証方式は OAuth2 を採用する。Service Account は個人 Gmail 直読の初期構成には適さないため不採用とする。資格情報はローカル開発では `credentials.json` とトークンキャッシュを `.gitignore` 管理し、GitHub 等へは絶対にコミットしない。Phase 2 以降に自動化基盤へ移す場合は secrets 管理へ移行する。

## Q05. 「未処理」の状態管理はどうするか？
**セクション**: 11.3
**質問**: 「まだ未処理である」メールを判定する仕組みとして、Gmailラベル・既読/未読フラグ・保存済みmessage_idのどれを使うか？
**回答**: 未処理判定は `data/raw/` に保存済みの `message_id` との突き合わせを主判定とする。Gmail ラベルは補助的に用いる。既読/未読フラグは人手操作で変化しやすいため正式判定には使わない。

## Q06. メール解析にLLMを使うか？
**セクション**: 12.3, 12.4
**質問**: Grokの出力が毎回完全に構造化されているとは限らない。LLM（Claude API等）を使った解析も選択肢に含めるか？
**回答**: Phase 1 は正規表現とルールベース解析を基本とする。メール書式の揺れが大きい場合に備え、Phase 2 以降で LLM 補助解析をオプション導入できる設計にする。初期実装で Claude API 等を必須依存にはしない。

## Q07. Title類似度の閾値と「直近数日」の定義は何か？
**セクション**: 14.2
**質問**: 「Title類似度が高い」の判定閾値はどこで定めるか？「直近数日」は何日か？
**回答**: 初期版では URL 完全一致を最優先とし、タイトル類似度は補助判定とする。タイトル類似度の暫定閾値は 0.85 相当とし、「直近数日」は 7 日と定義する。運用結果を見て調整する。

## Q08. 重複判定の参照データはどこに保存するか？
**セクション**: 14.2
**質問**: 重複判定に使う過去X URLの一覧はdata/index.jsonで管理するのか、それとも専用ファイルを設けるか？
**回答**: 重複判定用の過去 URL 一覧は `data/dedupe_index.json` のような専用ファイルで管理する。`data/index.json` は公開記事一覧に集中させる。長期化した場合は日付単位で分割または SQLite 化を検討する。

## Q09. 記事生成（編集コメント・総括文）には何のAIを使うか？
**セクション**: 15.6, 15.7
**質問**: 編集コメントと総括文の自動生成にはどのAI/LLMを使うか。初期段階ではルールベースか自動生成どちらを優先するか？
**回答**: Phase 1 では編集コメント・総括文はルールベース優先とし、必要に応じて対話型 AI を使って補助生成する。Phase 2 以降で API 利用を選択可能とするが、初期版では必須要件にしない。

## Q10. サイトのフレームワーク・テーマは何か？
**セクション**: 16.3, 17.1, 26（末尾追記）
**質問**: `site/_posts/` の構造はJekyll想定か？配信ブランチは？「CDNを使ったモダン技術ブログ的外観」の具体的なフレームワーク指定はあるか？
**回答**: `site/_posts/` 構成を採るため、初期版は Jekyll 前提とする。配信は GitHub Pages の標準構成に合わせ、main ブランチの Pages 設定を基本候補とする。デザインはまずシンプルな技術ブログ風に留め、CSS フレームワークは後段で決める。

## Q11. GoogleアドセンスはSPEC本文要件として扱うか？
**セクション**: 26（末尾追記）
**質問**: アドセンスのコードがSPEC末尾に記述されているが、これを正式要件とするか？初期公開時から承認済みか？
**回答**: アドセンスは初期公開の必須要件としない。別紙または将来要件として管理し、SPEC 本文の必須機能からは切り離す。承認状態が不明なため、実装優先順位は低とする。

## Q12. 記事の言語は日本語か？
**セクション**: 9.3（プロンプト末尾 "Output in Japanese"）
**質問**: Grokの出力は日本語指定だが、記事本文もすべて日本語か？英日混在の方針はあるか？
**回答**: 記事本文は原則として全文日本語とする。英語の固有名詞、製品名、引用元タイトルのみ必要に応じて併記する。英日混在を基本方針にはしない。

## Q13. GitHub Actionsや自動実行基盤を使うか？
**セクション**: 19.2, 19.3, 25（Phase 2以降）
**質問**: Gmail取得・記事生成・push・公開の自動化にGitHub ActionsやcronジョブなどのCI/CD基盤を使うか？secretsの管理方針は？
**回答**: Phase 1 はローカル実行＋手動 push を基本とする。Phase 2 以降で GitHub Actions を検討する。secrets は GitHub Actions Secrets を利用し、ローカル credential を直接リポジトリに置かない。

## Q14. エラー時の再試行の回数・間隔は？
**セクション**: 20.2
**質問**: 「再試行」とあるが、リトライ回数・間隔の定義がない。
**回答**: Gmail 取得の再試行は 3 回、30 秒間隔を初期値とする。公開処理失敗時は自動再試行せず、人間確認を優先する。

## Q15. インデックスロールバックの具体的な方法は？
**セクション**: 20.5
**質問**: 「インデックス更新をロールバック可能にする」の具体的な方式は何か？
**回答**: インデックス更新前にバックアップファイルを一時保存し、公開前の最終単位は Git コミットで管理する。初期版では `index.json.bak` による簡易ロールバックと Git による巻き戻しの併用とする。

## Q16. Phase 0の実施者とツールは何か？
**セクション**: 10.1〜10.6
**質問**: Phase 0の実証実験は、手動でGmailを確認するのか、`poc_gmail_read.py` スクリプトを実行するのか？実行環境の制約はあるか？

## Q19. 「CDNを使ったモダン技術ブログ的外観」の達成条件は何か？
**セクション**: 26（末尾追記）

**回答**: 「CDNを使ったモダン技術ブログ的外観」は初期版の正式要件にしない。初期版はシンプルで可読性の高いレイアウトを採用し、CSS フレームワークや CDN 利用は後段で決定する。

## Q20. AdSense導入に伴う透明性表示・ポリシー文言は必要か？
**セクション**: 18, 26（末尾追記）
**質問**: 収集・編集方針の透明性表示は定義されているが、第三者広告スクリプトを導入する場合の広告表示方針、プライバシー告知、必要な説明文の掲載場所が未定義である。これらも初期公開要件に含めるか？
**回答**: 広告導入時は、広告表示方針・プライバシー告知・必要な説明文を別途定義する。ただし初期版では広告を必須導入しないため、初期公開要件には含めない。導入時に `privacy` / `about` / `footer` 等への掲載方針を追加定義する。

## Q21. サイトのデザイン・レイアウトにおける構成要素は？
**セクション**: 17.3, 17.4
**質問**: 「シンプルな技術ブログ風」としているが、サイドバーの有無、ヘッダー/フッターの固定要素（検索窓、タグクラウド、SNSリンク等）についての具体的な希望はあるか？
**回答**: 初期版はサイドバーなしの1カラム構成またはシンプルな右サイドバー構成とし、ヘッダーにタイトルとナビゲーション（About/Categories）、フッターにポリシーと著作権表示を置く。検索機能やタグクラウドは記事数が増えた段階で検討する。

## Q22. X投稿内の画像やメディアの扱いはどうするか？
**セクション**: 5.3, 15.5
**質問**: SPECでは「要約とリンク中心」としているが、元投稿に画像や動画が含まれる場合、サムネイルとして表示するか、あるいは完全にテキストのみとするか？
**回答**: 初期版はテキストのみ（リンク含む）とする。画像の自動取得や埋め込みは著作権および実装コストの観点からPhase 2以降の検討事項とする。ただし、OGP画像が自動で表示されるような仕組み（リンクカード等）は許容する。

## Q23. 読者からのフィードバックやコメント機能は導入するか？
**セクション**: 17.4, 24
**質問**: 静的サイト（GitHub Pages）において、GitHub Issuesを利用したコメント欄（Giscus等）や、問い合わせフォームなどの設置予定はあるか？
**回答**: 初期版ではサイト内コメント機能やフォームは設置しない。フッターに SNS（X等）へのリンクを置くが、これは情報共有用であり、読者からの反応は X 上での言及（リプライや引用ポスト）を想定している。サイト内での直接的なインタラクション機能は将来的な拡張案とする。
